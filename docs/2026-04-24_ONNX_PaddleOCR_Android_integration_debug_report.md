## 発見した問題と修正 (5つ)

### 問題1: 正規化定数の不一致

**症状:** 出力の全タイムステップでclass 0 (blank) が90%以上。
**原因:** ImageNetの平均/標準偏差 `[103.5, 116.3, 123.7] / [57.4, 57.1, 58.4]` を使用していたが、PaddleOCRは `(pixel - 127.5) / 127.5` を期待している。
**修正:** `PADDLE_MEAN = 127.5f`, `PADDLE_STD = 127.5f` を全チャネル共通で適用。

---

### 問題2: チャンネル順序の不一致

**症状:** 修正1後もblank=40/40のまま。
**原因:** PaddleOCRネイティブはBGR入力だが、PNNX変換モデル(PyTorch由来)はRGB入力を期待する。当初BGRで実装していた。
**修正:** 平面0=R, 平面1=G, 平面2=Bの順で配置。

---

### 問題3: テストActivityでの画像事前変形

**症状:** `OnnxTestActivity.loadAssetBitmap()` で画像を320×48に事前stretchしていた。1536×2048のテスト画像が歪み、文字情報が破壊された。
**修正:** `BitmapFactory.decodeStream()` の結果をそのままEngineに渡す。アスペクト比維持はEngine側で処理。

---

### 問題4: アスペクト比を維持したリサイズの実装不在

**症状:** Engine側で単純に320×48にストレッチしていた。
**修正:** 
- 高さ48に固定、幅は画像のアスペクト比に応じて計算
- `resizedW = (srcW/srcH) × 48` (最大320)
- 幅が320に満たない場合、残りを0埋めでパディング

---

### 問題5: パディング値の不一致 ⭐ 最も重要な発見

**症状:** 修正1-4後もAndroidではblank=40/40のまま。Pythonでは同じ前処理で正しく動作。
**原因:** Androidの`FloatArray`はデフォルト値が`0.0f`。モデルはパディング領域が正規化後の黒(`(0-127.5)/127.5 = -1.0`)であることを期待している。`0.0`パディングはモデルにとって"灰色"のピクセルとして扱われ、特徴抽出層を通過して出力に影響した。
**検証方法:** Pythonで同一モデルに同一入力を渡し、パディング値のみを`-1.0`と`0.0`で比較:
  - `-1.0`パディング: blank=23/40, decoded='cYnaGetaYaYelowb'
  - `0.0`パディング: blank=39/40, decoded='C'
**修正:** `FloatArray(totalSize) { -1.0f }` で明示的に-1.0で初期化。

---

## 問題の根本原因の理解

PaddleOCRの認識モデルはCTC (Connectionist Temporal Classification)ベース。最終層の線形変換は `logits = W @ features + b` で計算される。

パディング値が`0.0`(灰色)の場合、特徴マップに余分な活性化が生じ、CNN層を通じて空白トークンへのバイアスが強化される。一方、`-1.0`(黒、正規化後の背景)はモデルが「空白領域」として認識し、正しくblankを出力する。これがCTCデコーダーにおいて非空白文字が正しく認識されるために不可欠だった。

## 実装で気をつけるべき点

### 1. 正規化はモデルに合わせる

```
PaddleOCR: (pixel - 127.5) / 127.5    // mean=0.5, std=0.5 on [0,1]
ImageNet:  (pixel - [103.5,116.3,123.7]) / [57.4,57.1,58.4]  // 違う!
```

モデルの学習時の前処理を必ず確認すること。ONNXモデルのメタ情報だけでは正規化方法はわからない。

### 2. パディング値は正規化後の黒で埋める

画像を0パディングした場合、正規化後は`(0-127.5)/127.5 = -1.0`になる。テンソルを直接構築する場合は**正規化前の0パディング → 正規化後の-1.0に相当する**ことを意識する。

```kotlin
// NG: デフォルト0埋め
val arr = FloatArray(size)  // 0.0

// OK: 正規化後の黒(-1.0)で埋める
val arr = FloatArray(size) { -1.0f }
```

### 3. チャンネル順序は変換元フレームワークに依存

- PaddlePaddle → BGR
- PyTorch → RGB
- PNNX変換 → RGB (PyTorch由来)
- ONNX標準 → チャンネル順はモデル依存

`session.inputInfo`で確認するか、PythonでBGR/RGR両方を試す。

### 4. アスペクト比はモデルの入力仕様に合わせる

PaddleOCR認識モデルは可変幅入力(`shape=[-1, 3, 48, -1]`)。高さ48固定、幅は比率に応じて計算し、最大320でパディング。

### 5. JPEGのEXIF orientationに注意

`BitmapFactory.decodeStream()`はEXIF orientationを**適用しない場合がある**(エミュレーター確認済み)。`ExifInterface`で明示的に確認・回転処理が必要。本番ではカメラ画像の向きを常に確認すること。

### 6. Pythonで先に検証する

ONNXモデルはプラットフォーム非依存のはずだが、ONNX Runtimeの実装依存の差異や前処理の違いで結果が変わる。同じモデル・画像・前処理でPython → Androidと比較検証するのが最も確実なデバッグ方法。

## 教訓

1. **モデルの前処理仕様を最初に確認する** — 正規化、チャンネル順序、リサイズ、パディング。ONNXモデルファイル自体にはこの情報が含まれない。
2. **パディング値のような「些細な」パラメータが致命的な影響を与える** — `-1.0` vs `0.0` の差だけでOCR結果が空から完全な文字列に変わった。
3. **Pythonは強力な検証ツール** — Androidで悩む前に、Pythonのonnxruntimeで同じモデルを動かせば、モデルの問題かプラットフォームの問題かを即座に切り分けられる。
4. **テスト画像の選択** — 1536×2048のポートレート画像を36px幅に圧縮してテストするのは適切ではなかった。テキストが横に並ぶランドスケープ画像を使うべき。
